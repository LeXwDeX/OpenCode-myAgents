# AHE 提示词优化器 — 编码 Agent 文本组件与代码相关文本优化

基于 AHE (Agentic Harness Engineering) 方法论，对编码 Agent（Claude Code / opencode / Cursor agent / Cline 等）的文本组件进行诊断和优化。适用于一切由提示词、子 agent 定义、工具描述、skill、记忆文件和代码相关文本塑造行为的 prompt-driven 编码工具。代码相关文本包括注释、docstring、commit message、debug 说明和代码说明文档；优化目标不是“写得更像人”，而是砍掉废话，留下事实、约束、证据和可检索记忆。

---

## 1. 核心原则

1. **删行为指导，留硬约束** — "你应该先查文档" ❌ → "遇到未知 API 返回 error，不要猜测参数名" ✅。行为指导僵化进化空间；硬约束可证伪、可审计。
2. **证据驱动** — 每次修改必须追溯到一个具体失败案例。没有案例不做编辑。
3. **正确组件层级** — 一条修复只放一个位置。堆规则到根提示词是最常见的错误（§5 决策表）。
4. **可证伪预测** — 每个 `chg-N` 写预测"哪些行为翻转"。下次验证不成立则回滚，而非加强。

---

## 2. 组件分类法

编辑前强制把 agent 的提示词表面分解到以下层级。每个修复必须落在且仅落在其中一层。

| 层级 | 存放内容 | 何时编辑 |
|------|---------|---------|
| **角色 / 身份** | agent 是谁，1–2 句话 | 自我认知越权/矮化/语气错 |
| **环境契约** | 可用工具、文件布局、不可假设的东西 | 幻觉路径/工具/能力 |
| **通用规则** | 适用所有任务的硬约束 | 同一约束跨任务类型被违反 |
| **工作流指导** | 步骤化程序 + 硬门禁 | 跳步/顺序错/过早终止 |
| **工具使用规则** | 何时调哪个工具、错误处理、分页 | 误用特定工具 |
| **工具描述**（逐工具） | 贴身指导 | 坑是工具特有的 |
| **输出契约** | 交付物精确形状 | 输出不一致/下游解析崩 |
| **示例 / few-shot** | 锚定边界的 worked example | 小众 case 反复出错 |
| **子 agent 提示词** | 隔离上下文工作者的提示词 + 输出契约 | 行为够专一，塞主上下文会膨胀 |
| **Skill**（按需加载） | 仅相关时加载的 playbook | 只有部分任务需要的可复用模式 |
| **代码相关文本** | AI 撰写或维护的注释、docstring、commit message、debug 说明、代码说明文档 | 文本在解释“代码显然做了什么”而非记录“为什么必须这样 / 哪个事实要求这样” |
| **长期记忆** | 跨会话持久知识（不绑具体插件名） | 每次会话重学同一课 |
| **短期记忆** | 会话级 plan/findings/progress | `/clear` 后丢计划 |
| **代码库即记忆** | git history / blame / 行内注释 / 日志 / 代码本身 | 重新调试 commit log 早已解释的 bug |
| **纠错规则** | 三振出局 / 回滚 / 上报 | 循环同一错误 / 不停 / 不问 |
| **记忆-写 hooks** | 何时写入、写到哪个面 | 记忆为空或充满噪声 |

## 3. 六维检查表

对目标文本逐段过以下六维，任一命中即扣分：

| 检查项 | 扣分信号 | 扣分原因 |
|--------|---------|---------|
| **行为指导** | "你应该"、"建议"、"最好"、"优先"、"记住要" | 僵化进化空间，模型自己会找最优策略 |
| **冗余解释** | 解释工具用法、LLM 已知常识 | 浪费 context token，挤掉任务空间 |
| **模糊约束** | "尽量"、"适当"、"合理" | 不可证伪，无法判断是否生效 |
| **交叉覆盖** | 同一规则出现在多个地方 | 修改时遗漏产生不一致 |
| **无证据指令** | 看不出防止什么失败；注释/说明/commit message 没有指向代码、日志、trace、测试、提交记录、debug 记录或用户反馈 | 无法判断是否可删，也不能作为后续 agent 可验证、可检索的记忆 |
| **过时假设** | 针对旧版模型/工具的限制 | 模型升级后变负面约束 |

## 4. 诊断流程：evaluate → analyze → improve

### Step 1 · 评估

端到端读完全部提示词表面（根提示词、子 agent、工具描述、skill、记忆文件、代码相关文本）。写评估笔记：
- agent 应该做什么（用你的话）。
- 做得好的。
- 分类法中哪些层级缺失 / 贫血 / 臃肿。
- 哪里跨层矛盾。
- **记忆和纠错是否缺失** —— 最常见缺口。
- 代码相关文本是否有事实来源：代码、日志、trace、测试结果、commit history、debug 注释或用户反馈。

### Step 2 · 分析 —— 归组为模式类

将症状归入 **3–4 个失败模式类**（不是列十个 case）。每类写：

```
模式 P<n>：<名称>
- 证据    ：case / 引用
- 根因    ：机制（为什么发生，1–2 句）
- 组件层级：哪层拥有修复权
```

要机制不要症状。差："agent 循环"。好："无停止规则；verify 失败时重试同一修复，因为没有'两次失败后质疑假设'指令。修复属于纠错规则层。"

### Step 3 · 修复 —— 四字段变更清单

每个编辑必须带齐四字段，缺一不做：

```
chg-N · <一行描述>
├── 失败证据   ：哪些 case 驱动 —— 引用
├── 根因       ：机制，1–2 句
├── 针对性修复 ：最小编辑
├── 预测影响   ：翻转 → [……]；风险 → [……]
└── 组件层级   ：<角色/环境/通用/工作流/工具使用/工具描述/输出/示例/子agent/skill/代码相关文本/长期记忆/短期记忆/代码库记忆/纠错/记忆-写hooks>
```

## 5. 选对组件层级 —— 决策表

| 症状 | 正确层级 |
|------|---------|
| 所有任务类型违反同一约束 | **通用规则** |
| 步骤对但顺序错 / 跳步 | **工作流指导** |
| 误用特定工具 | **工具使用规则 / 工具描述** |
| 输出形状不一致 | **输出契约** |
| 90% 好，小众 case 出错 | **示例** |
| 只有特定模式需要 | **Skill** |
| 足够专一值得隔离 | **子 agent** |
| 注释 / commit message / debug 说明只复述代码，没有解释约束来源 | **代码相关文本** —— 删除“做了什么”的废话，保留“为什么必须这样”的事实来源 |
| 每次重学同一课 | **长期记忆** |
| commit log 早已解释但 agent 重新调试 / 凭想象改代码 | **工具使用规则 + 记忆-读 hooks**（§7） |
| `/clear` 后丢计划 | **短期记忆纪律** |
| 循环错误 / 不上报 / 不停 | **纠错规则** |
| 记忆为空或噪声 | **记忆-写 hooks** |
| 同层修两次无效 | **换层** —— 反复加强同一规则不如换层 |

## 6. 实战模式

### 模式 A：有失败案例

用户提供当前文本 + 失败场景 → 严格按 Step 2–3 输出，每个修改追溯到具体失败。

### 模式 B：纯审计（无失败案例）

用户只给文本 → 跑六维检查表，标注所有扣分点，输出"哪些指令没证据支撑、建议下次失败时验证删除"。

### 模式 C：对比优化

用户给旧版 + 新版文本 → diff 对比，用核心原则评判每处修改：
- ✅ 删行为指导 → 加分
- ❌ 新增行为指导 → 扣分
- ✅ 模糊→具体 → 加分
- ❌ 无证据新增规则 → 扣分

## 7. 记忆是复数面

健壮的编码 agent 没有单一记忆，有一**组记忆面**。提示词系统的职责：让 agent 知道每个面存什么、行动前读对的面、行动后写对的面。

**六个面**（按角色描述，绝不写产品名）：

| 面 | 读时机 | 写时机 |
|----|--------|--------|
| **持久 prompt 侧记忆**（用户环境中已接入的记忆插件） | 任务开始查询相关条目 | 任务结束写决策摘要；发现坑写原子事实 |
| **短期 scratch** | 会话开始恢复 plan | 每个 checkpoint 更新 |
| **Git history** | 调试前 `git log -p -- <file>` / `git blame` | 提交时写精确 commit message |
| **行内注释** | 编辑前读周围 `# why:` | 修非显然 bug 时留 `# why:` |
| **运行日志** | 调查间歇性问题前 grep logs | —— |
| **代码本身** | 编辑前重读目标行（不凭想象改） | —— |

**关键指令：** 撰写被优化 agent 提示词中的记忆章节时，**必须主动指示 agent 使用其环境中可用的持久记忆插件**——任务开始查询、任务结束写入、发现坑写入。假设已有插件接入；指示 agent 发现并使用。**不写出插件名称、文件路径或厂商绑定的调用示例。**

## 8. 纠错规则

缺失纠错几乎总是第二高影响的 `chg-N`。要编码的模式：

- **三振出局** — 同一错误三次 → 停、质疑假设、上报用户。
- **Verify 是硬门禁** — 失败触发重规划或上报，不是绕过。
- **区分实现错误与 spec 错误** — verify 失败时考虑 spec 本身是否有误。
- **上报触发器** — 具体化：需求模糊、破坏性操作、反复失败、范围蔓延。
- **回滚纪律** — chg-N 错了就撤销，不在上面堆 chg-N+1 补偿。

## 9. 典型优化速查

| 问题 | 改前 | 改后 |
|------|------|------|
| 行为指导 | "你应该在修改代码前先读文件" | "文件修改工具拒绝空文件参数"（硬约束） |
| 模糊约束 | "尽量保持回复简洁" | （删除——模型自己会根据 context 调节） |
| 冗余解释 | "Git 是一个版本控制工具..." | （删除——LLM 已知 Git） |
| 注释废话 | `# increment i` | （删除——代码已说明做了什么） |
| Debug 注释 | `# workaround for timing issue` | `# why: service emits ready before socket binds; see failing log 2026-05-21` |
| Commit message | `fix bug` | `fix readiness race observed in startup log; wait for bound socket` |
| 交叉覆盖 | prompt 和 skill 都写了"用 rtk 前缀" | 只在 skill 写，prompt 引一句即可 |
| 过时假设 | "Claude 3.5 不支持..."（针对旧模型） | （删除——当前模型已支持） |
| 无记忆层 | agent 每次重学 snake_case 惯例 | 建长期记忆 + 记忆-写 hook；指示 agent 任务开始查插件 |

## 10. 反面案例：api-failure-protocol

**初版**（被纠正的版本）：
```
# API 失败处理协议
## 1. API 函数不存在 → 查文档
## 2. SSL 证书错误 → 跳过验证
## 3. 网络不可达 → 用替代源
## 4. 参数错误 → 看签名
```

**问题**：把"遇到不会的先查资料"这个通用原则，窄化成了 API 四步流程（行为指导）。

**纠正后**：
```
# 遇到不会的 → 先查资料
不管什么领域——API、SSL、语法、网络、配置——
不知道就是不知道，查了才知道。
```
→ 删四步流程（行为指导），留核心原则（硬约束）。

## 11. 输出契约

每次优化会话结束交付两件：

1. **修订后制品** — 全文或 diff，按改动文件分拆。
2. **evolve_summary 报告**：

```markdown
# AHE 提示词修订 — <agent 名称>

## 1. 范围与职责
<一行>

## 2. 组件地图（修改前）
<哪些层存在/缺失/贫血/臃肿>

## 3. 评估笔记
<2–4 行>

## 4. 失败模式类
### P1 — <名称>
- 证据 / 根因 / 组件层级

## 5. 变更清单
### chg-1 — <描述>
- 失败证据 / 根因 / 修复 / 预测 / 层级

## 6. 证伪计划
- 下次应重跑的 case
- 回归时先回滚哪个 chg-N
- 未处理的模式及原因
```

这份报告**就是**循环的记忆。没有它下次迭代无法证伪。

---

## 12. 写作风格

- 祈使语气 > 被动/期望式。
- 解释 why（简短）> 堆 MUST。
- Worked example > 抽象规则。
- 精简——不拉重量的句子删掉。
- 按主题分组，不按编辑时间线。
- 记忆条目是原子事实 + 源案例，不是段落建议。

---

## 13. 证伪 —— 交错迭代

每次修订是一个预测。告诉用户：

> "下次跑这些 case，如果预测不成立，回滚对应 chg-N 而非加强它。基于实际翻转修订，不基于感觉。"

鼓励每次修订作为单独 git commit，标注预测翻转。这是 AHE `runs/iteration_NNN/` 纪律的文件系统类比。

---

## 反模式清单

- 无失败案例就动手改。
- 列十个 case 而非归三四个模式类。
- 同一规则第三次加强而非换层。
- 堆全大写 MUST 代替解释机制。
- 留长期记忆层空白。
- 把记忆当单一文件而非一组面。
- 记忆章节只描述存在而非指示 agent 主动使用插件。
- 绑定具体记忆插件名/路径/调用形状。
- 留纠错隐式。
- 系统越改越长（应在常驻 prompt 净减、在记忆/skill/子 agent 净增）。
- 会话事实写入长期记忆。
- 跳过 evolve_summary 报告。
- 新增行为指导（"你应该……"）而非硬约束。
- 把通用原则窄化成特定流程（api-failure-protocol 反面案例）。
